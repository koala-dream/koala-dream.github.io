问题1：主引导程序不能超过512字节

突破方法：在主引导程序中，加载更大的程序到某一个地址，在主引导程序结束之前跳转过去

具体要怎么做呢？

1. 首先必须要引入文件系统，只有通过文件系统，才能通过文件名找到文件，以及知道目标文件的大小
2. 将创建好的软盘格式化成FAT12文件系统格式
3. 将格式化后的软盘挂载到Linux文件系统下，把编译好的loader.bin程序放进去

第五课：主引导程序的扩展（下）

问题1：FAT12文件系统里的文件，数据存放不一定是连续的，那要怎么获得完整的文件数据？

方法：通过FAT表来查找，FAT表每一项记录着下一个应该读取的扇区地址，最后一项标记为特殊项，表示结束

流程：假设一个文件的起始扇区在39扇区，它的FAT表项在2+6=8索引处，操作系统应该先读39扇区内容，读完之后去FAT表的第8项查看是不是为最后一项，如果不是，就读这第8项里面的信息，读出来如果是12，则表示操作系统应该去起始簇号为12的位置读数据，对应的物理起始地址为（0x4200 + （12-2）*512 ），读完之后再去FAT表查看第12项是不是为最后一项，周而复始进行这个过程，直到读完为止

必须知道的知识1：

  1. 在FAT12文件系统下，第1个扇区是主引导程序，第19个扇区是文件目录项扇区

  2. FAT12目录区里，存放着多个目录项，每个目录项由32个字节组成（关键点是要里面的名称，起始地址，文件大小）

  3. 在FAT12文件系统里的文件，数据存放不一定是连续的，一个文件可能分多个扇区存储

  4. FAT表中每一个项占用12bits，表项值大于等于0xFF8，说明已经达到最后一个扇区（一个簇）

  5. FAT12文件系统中，一个扇区大小等于一个簇的大小（通常来说簇的概念比扇区更大）

  6. 数据区起始扇区号为33，地址为0x4200，对应的FAT表的表项编号为2（不为0）：打个比方，一个文件的起始扇区在39扇区，它的FAT表项在2+6=8索引处。

  7. DIR_FstClus表示FAT表项的起始位置（起始簇号，不是扇区号），每一个文件都有一个DIR_FstClus

  8. 所以，DIR_FstCIus对应的物理起始地址为（0x4200 + （DIR_FstClus-2）*512 ）

可以用互斥锁、信号量，不能用自旋锁

互斥锁的设计：不会关中断，会让尝试获得锁的应用程序陷入休眠状态（放入等待队列）。当锁被释放后，会唤醒等待队列中的进程

自旋锁的设计：可能会关中断（不允许进程休眠），用于保护非常短的临界区

第一步

抛出问题：插入Linux板卡的时候，第一个和他通信的驱动程序是谁

答案：主机控制器驱动（例如 ehci-hcd、xhci-hcd）

职责

  • 复位设备：向新插入的设备发送复位信号。
  • 分配地址：通过控制传输为设备分配一个唯一的 USB 地址（通常是地址 1，然后重新分配）。
  • 获取设备描述符：读取设备的基本信息（如厂商 ID、产品 ID、设备类等）。
  • 这些操作都是在主机控制器驱动的控制下，通过 USB 总线发送数据包完成的。

第二步：

USB 核心（USB Core）的介入

  • 主机控制器驱动将获取到的信息上报给 USB 核心（USB Core，即内核中的 usbcore 模块）。
  • USB 核心维护着所有已注册的 USB 设备驱动（如 usbhid、usbkbd）以及它们的 usb_device_id 表。
  • USB 核心开始枚举过程：读取设备的配置、接口、端点等信息，并创建一个 usb_interface 代表键盘的接口。

第三步：

匹配具体的设备驱动

  • USB 核心遍历所有已注册的 USB 设备驱动的 id_table，寻找与当前设备匹配的条目。

  • 对于标准 USB 键盘，通常匹配的是 HID 类驱动（如 usbhid 或你之前看到的 usbkbd）。

  • 一旦找到匹配，USB 核心调用该驱动的 probe 函数，并将 usb_interface 交给它。

  • 设备驱动（如 usbkbd）在 probe 中初始化设备，注册输入设备，并提交 URB 开始接收键盘数据。

  • 至此，键盘才真正可以被用户空间使用。

总结

  • 第一个与硬件通信的是主机控制器驱动，它负责最底层的硬件交互和初始化。
  • 第二个参与的是 USB 核心，它负责管理枚举和驱动匹配。
  • 最终具体功能由专门的 USB 设备驱动（如 HID 驱动）实现。

四、注意事项（避坑指南）

波特率必须一致（500k）

Linux端和STM32端都要设置成完全相同的波特率（500k）。

终端电阻一定要加

两个120Ω电阻并联在总线上

GND连接

若STM32和RK3568使用独立电源，必须共地（接GND），否则无法通信。

数据越界处理

STM32侧收到>200的数值时，建议直接忽略或按200处理，防止数组溢出。

五、扩展（未来需要更多功能）

  • 0x103：喇叭/蜂鸣器
  • 0x104：参数配置

每个ID固定1字节数据，STM32端加一个case即可，完全不影响原有控制逻辑。

为什么要用 perror？

系统调用（如 socket、bind、ioctl）失败时，内核会设置一个全局错误码 errno（例如 13 = EACCES）。
perror 会自动读取当前的 errno，把它翻译成人类可读的字符串并打印。

自定义函数名核心原则

  1. 全局符号加前缀（避免链接冲突）
    • 格式：模块名_动作_对象
    • 例如：can_init, uart_send_byte, gpio_set_level
  2. 动词开头 + 宾语（一眼看出功能）
    • 例如：get_temperature, set_motor_speed, enable_interrupt
  3. 统一风格，全小写 + 下划线（snake_case）
    • C 标准库、Linux 内核都是这种风格，最符合 C 语言传统。
  4. 私有/静态函数可加下划线前缀
    • 例如：_can_write_reg, _calc_crc16
