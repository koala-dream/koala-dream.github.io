1. 帧格式

  • 标准帧（11位ID）
  • 数据长度：固定1字节（DLC = 1）

2. ID分配（两个ID就够了）

CAN ID 功能 数据字节含义
0x100 油门控制 0=停止；1100=前进速度百分比；101200=后退速度百分比
0x101 转向控制 0=直行；1100=左转角度百分比；101200=右转角度百分比

一个ID管速度，一个ID管方向，STM32收到后直接查表映射到PWM，无需解析命令码

3. 数据映射示例

油门控制（ID=0x100）

发送字节 实际动作
0 电机停止
50 50%速度前进
100 100%速度前进
150 50%速度后退
200 100%速度后退

转向控制（ID=0x101）

发送字节 实际动作
0 车轮回正
50 左转50%角度
100 左转最大角度
150 右转50%角度
200 右转最大角度

四、注意事项（避坑指南）

波特率必须一致（500k）

Linux端和STM32端都要设置成完全相同的波特率（500k）。

终端电阻一定要加

两个120Ω电阻并联在总线上

GND连接

若STM32和RK3568使用独立电源，必须共地（接GND），否则无法通信。

数据越界处理

STM32侧收到>200的数值时，建议直接忽略或按200处理，防止数组溢出。

五、扩展（未来需要更多功能）

按此极简方案，未来增加功能只需新增ID，例如：

  • 0x102：灯光控制
  • 0x103：喇叭/蜂鸣器
  • 0x104：参数配置

每个ID固定1字节数据，STM32端加一个case即可，完全不影响原有控制逻辑。

为什么要用 perror？

系统调用（如 socket、bind、ioctl）失败时，内核会设置一个全局错误码 errno（例如 13 = EACCES）。
perror 会自动读取当前的 errno，把它翻译成人类可读的字符串并打印。

自定义函数名核心原则

  1. 全局符号加前缀（避免链接冲突）
    • 格式：模块名_动作_对象
    • 例如：can_init, uart_send_byte, gpio_set_level
  2. 动词开头 + 宾语（一眼看出功能）
    • 例如：get_temperature, set_motor_speed, enable_interrupt
  3. 统一风格，全小写 + 下划线（snake_case）
    • C 标准库、Linux 内核都是这种风格，最符合 C 语言传统。
  4. 私有/静态函数可加下划线前缀
    • 例如：_can_write_reg, _calc_crc16
